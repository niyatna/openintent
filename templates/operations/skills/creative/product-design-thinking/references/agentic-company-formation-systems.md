# Agentic Company Formation Systems — Niyatna Pattern

Use this reference when the user is framing products like Niyatna, OpenIntent, Proof of Intent, Claude Tag comparisons, or any company-wide agent operating layer.

## Core distinction

Do not collapse an agentic-company formation system into “AI assistant in workspace.” The durable pattern is:

- **Workspace AI / taggable assistant:** an AI teammate can be invoked inside Slack/Discord/docs to perform tasks.
- **Agentic company formation system:** the company gains a governed operating layer: intent packets, role boundaries, owned work, memory, agent workforce, routing, automations, proof/evidence, and learning loops.

A concise lens:

> A normal company becomes agentic when it becomes askable, assignable, remembered, automated, and proven.

## Niyatna vocabulary map

- **Niyatna** — parent formation system / operating standard.
- **Agentic Company** — category: company where people, agents, memory, permissions, recurring work, and proof operate as one governed system.
- **OpenIntent** — protocol/spec/kit for turning human/company requests into structured intent packets.
- **Proof of Intent** — verification primitive linking original intent to routing, execution, evidence, approval, memory update, and report-back.
- **Niyatna OS** — company-wide permissioned channel/operating layer.
- **Niyatna HQ / Command Room** — control surface where intent becomes owned work.
- **Niyatna Agents** — workforce units with mission, scope, tools, memory, boundaries, proof duties, and handoff rules.
- **Niyatna Route** — routing layer across model/tool/human/agent paths based on sensitivity, cost, latency, tools, risk, and fallback.
- **Niyatna Memory** — company memory: canon, SOPs, decisions, customers/vendors, proof logs, approval history, execution records.
- **Niyatna Automations** — scheduled/event-driven recurring work.

## Product/design implications

When analyzing or explaining this class of product, work from foundations upward:

1. **Domain layer:** define the real domain objects — Intent, Agent, Human Role, Department, Permission Boundary, Task/Owned Work, Route, Proof, Memory Write, Approval, Report-back.
2. **User needs:** users are not only founders. Include authorized employees, department leads, auditors, partners, selected clients, and agents inside scoped boundaries.
3. **Product strategy:** scarce value is not access to AI tools; it is organizing AI capability into a trustworthy company system.
4. **Conceptual model:** agent identity is not a gimmick if it is tied to mission, memory, tools, boundaries, channel presence, and proof duties.
5. **Interaction flow:** every request should become traceable: request → intent packet → role/permission check → assignment → route/execution → evidence → approval/evaluation → memory write → report-back.
6. **Surface:** avoid tool-collage framing. Public copy should say system, operating layer, formation stack, command room, proof, and standards — not “we have many AI tools.”

## Pitfalls to avoid

- Framing as “AI automation agency,” “generic SaaS,” “AI tools for teams,” “chatbot wrapper,” or “Slack bot.”
- Treating company AI as everyone individually using ChatGPT/Claude.
- Treating the owner dashboard as the whole product; agentic company is company-wide and permissioned.
- Showing raw implementation screenshots before the category is understood.
- Overclaiming security or proof. Prefer scoped language: designed for, approval-gated, permissioned, where configured.
- Ignoring memory ownership and data boundaries; those are central to the category.

## Canon lines worth preserving

- “Tools are organs. Niyatna is the body plan, nervous system, law, and command room that make those organs operate as a company.”
- “The agents are not the company. The intent is the company. The agents are the leverage.”
- “When anyone can solve problems with AI, the scarce thing becomes the operating system to organize it into a company.”
- “Memory is infrastructure, not vibes.”
- “Company data is not casual prompt material.”
