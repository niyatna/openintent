# Niyatna OS - OpenIntent Kit Development Standards

This document establishes the official developer rules, conventions, and acceptance criteria for any configuration, template, or agent skill introduced to the OpenIntent Kit multi-agent company orchestrator.

---

## 1. Zero Personal Branding & Intimate Personas

To prepare the multi-agent stack as a clean, industry-grade corporate starter kit for remote VPS deployments, all personal, customized, or intimate names, aliases, and contacts are strictly prohibited from the repository code, configurations, memory baseline files, and skill templates.

### Banned Entities:
- **Banned Personal Names**: `Muhamad Galih Saputra`, `Galih`, `Galih's`, `galih`.
- **Banned Intimate Personas**: `Keiya`, `Keiya Putri Zeyni`, `Keiya's`, `keiya`, `galyarder` (when used as personal developer alias).
- **Banned Private Emails**: `keiyazeyniputri@gmail.com`, `galyarderlabs@gmail.com`, `galyarder@gmail.com`.
- **Banned Creator Logos/Orgs**: `Galyarder Labs`, `Galyarder`.

### Allowed Replacements:
- **User Reference**: `User`, `Owner`, `Founder`, `Investor` (depending on the profile).
- **Agent Reference**: `Operations Agent` (for default), `Corporate Agent` (for corporate), `Public Agent` (for public), or generic `Agent`.
- **Corporate Organization**: `Company`, `Organization`, `Company Operations` (under author keys).
- **Corporate Contacts**: `admin@company.com`, `support@company.com`.
- **Identity Tone**: Clean, professional, and business-focused. Professional `aku-kamu` is allowed only for internal agent-to-user team communications. No intimate, genit, or informal conversational slang.

## 2. Directory & Path Portability

The target VPS environment runs everything inside isolated Docker containers using volume mount bindings. Hardcoded absolute paths specific to the developer's local machine environment will fail immediately when container services start.

### Banned Paths:
- **Developer Local Home**: `/home/galyarder/`
- **Developer Local configuration**: `/home/galyarder/.hermes/`
- **Specific Workspace paths**: `/home/galyarder/projects/openintent/workspace`

### Allowed Paths:
- **System relative directories**: `./profiles/default`, `./profiles/corporate-agent`, `./profiles/public-agent`.
- **Container volume mappings**: `/opt/data/...` (which maps to root default profile volume `/opt/data/company-brain/`, `/opt/data/profiles/corporate-agent/corporate-brain/`, or `/opt/data/profiles/public-agent/public-brain/`).
- **User home abstraction**: `~/.hermes/` (acceptable fallback for local host auditing).

## 3. Role & Permission Isolation (Airlock Doctrine)

Niyatna OS segregates multi-agent responsibilities into separate containers and execution lanes to enforce privacy boundaries and prevent credential leaks.

### Profile Alignment:
1. **Default Profile (Operations Agent - Port 8001)**:
   - *Role*: Manages local files, repository checkouts, verify audits, and internal task queues.
   - *Security*: Open to all internal workers. Allowed to execute local system checks, prohibited from external public channels.
2. **Corporate Profile (Corporate Agent - Port 8002)**:
   - *Role*: Strategy, ROI forecasts, Runway budgeting, and security approval gates.
   - *Security*: Restricted to founders, board members, and C-level executives. Contains sensitive company metrics.
3. **Public Profile (Public Agent - Port 8003)**:
   - *Role*: Customer Support helper, FAQs, competitor analysis, public web searches.
   - *Security*: Sandboxed from company source codes, local files, and private databases.

### Memory Partition:
- Multi-agent memory banks inside `hindsight/config.json` must use dedicated `bankId` and `bank_id` configurations matching the profile name (`default`, `corporate`, `public`) to prevent memory leak/bleed across divisions.

## 4. Container-Native Execution & Environment Seeding

To ensure a seamless, `"1-minute ready"` bootstrap execution:
- **No Host Dependencies**: Tools or CLI binaries (`notebooklm-mcp-cli`, `paperclip-mcp`, `codegraph`, `officecli`, `claude-code`) must be installed internally inside the container environment using NPM global installer directives or PIP pip-wheels in the container entrypoint.
- **Auto-Generated Credentials**: Security credentials like basic auth `DASHBOARD_USERNAME`, `DASHBOARD_PASSWORD`, and `DASHBOARD_SECRET` must be automatically generated by `setup.sh` via cryptographic utilities (`openssl/urandom`) and directly injected into the staging `.env` templates.

---

## 5. Rules of Engagement for Coding Agents

- **No Local Runs**: Never run `docker compose up`, `docker compose up -d`, `install.sh`, `setup.sh`, or execute docker status/run script files locally on the development workspace. You verify by dry-running syntax checkers, re-diffing configs, and git file checks.
- **Atomic Commits**: Commit changes with conventional commit tags (`feat:`, `fix:`, `clean:`, `docs:`) and push to remote target branches cleanly.
